# Reporte de Mejoras — 2026-04-23

Auditoría automática del codebase de SEO Crawler (crawlsite.app). Se aplicaron 12 mejoras concretas agrupadas en cuatro categorías: **Seguridad**, **SEO**, **Flujo** y **Layout/Estructura**.

---

## Seguridad

### 1. Rate limiting en rutas de autenticación (`src/server.js`)

**Problema:** Solo el endpoint `/api/crawl` tenía rate limiting. Las rutas de login, registro, forgot-password y resend-verification eran vulnerables a fuerza bruta y credential stuffing.

**Solución:** Se creó `authLimiter` (20 intentos fallidos por IP en 15 minutos) y se aplicó a:
- `POST /api/auth/login`
- `POST /api/auth/register`
- `POST /api/auth/forgot-password`
- `POST /api/auth/resend-verification`

```js
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 20,
  standardHeaders: true,
  legacyHeaders: false,
  message: { error: "Demasiados intentos. Intenta de nuevo en 15 minutos." },
  skipSuccessfulRequests: true,
});
```

### 2. Corrección de Open Redirect en login (`pages/login.jsx`)

**Problema:** El parámetro `?next=` se usaba directamente sin validación. Un atacante podía construir una URL como `/login?next=https://sitio-malicioso.com` para redirigir usuarios tras autenticarse.

**Solución:** Se añadió la función `safeRedirect()` que valida que el destino sea una ruta interna relativa (empieza con `/`) y rechaza URLs absolutas o externas.

```js
function safeRedirect(next, fallback = "/projects") {
  if (typeof next !== "string" || !next) return fallback;
  try {
    const parsed = new URL(next, "http://x");
    if (parsed.origin !== "http://x") return fallback;
  } catch { return fallback; }
  return next.startsWith("/") ? next : fallback;
}
```

### 3. Security Headers HTTP en Next.js (`next.config.js`)

**Problema:** Las respuestas del servidor Next.js no incluían headers de seguridad estándar.

**Solución:** Se agregó el bloque `headers()` al `nextConfig` con:
- `X-Content-Type-Options: nosniff` — previene MIME sniffing
- `X-Frame-Options: SAMEORIGIN` — previene clickjacking
- `Referrer-Policy: strict-origin-when-cross-origin` — controla referrer
- `Permissions-Policy` — deshabilita cámara, micrófono y geolocalización
- `Strict-Transport-Security` — fuerza HTTPS (max-age 2 años + preload)
- `X-XSS-Protection: 1; mode=block` — protección XSS en browsers viejos

## SEO

### 4. `noindex, nofollow` en páginas de autenticación

**Problema:** Las páginas `/login` y `/register` eran indexables por motores de búsqueda, desperdiciando crawl budget y potencialmente apareciendo en resultados irrelevantes.

**Solución:** Se agregó `<meta name="robots" content="noindex, nofollow" />` en `pages/login.jsx` y `pages/register.jsx`.

### 5. Actualización de `robots.txt`

**Problema:** El archivo solo bloqueaba rutas del dashboard interno, pero dejaba accesibles páginas de autenticación y recuperación de cuenta.

**Solución:** Se añadieron las siguientes rutas al `Disallow`:
- `/login`
- `/register`
- `/forgot-password`
- `/reset-password`
- `/verify-email`
- `/subscription`

### 6. FAQPage Schema en `/precios`

**Problema:** La sección de preguntas frecuentes existía en HTML pero no en structured data, perdiendo la oportunidad de aparecer como FAQ rich result en Google.

**Solución:** Se agregó `@type: FAQPage` al array `@graph` del schema JSON-LD en `pages/precios.jsx` con las 4 preguntas ya presentes en la página.

### 7. `twitter:site` en páginas públicas

**Problema:** Faltaba el meta `twitter:site` en index y precios, reduciendo la calidad de las Twitter Cards cuando se comparten links.

**Solución:** Se agregó `<meta name="twitter:site" content="@aionsite" />` en `pages/index.jsx` y `pages/precios.jsx`.

### 8. OG tags en `/aviso-privacidad`

**Problema:** La página de aviso de privacidad carecía de meta tags Open Graph, mostrando un preview genérico al compartirla en redes sociales.

**Solución:** Se añadieron `og:type`, `og:url`, `og:title`, `og:description` y `og:image` en `pages/aviso-privacidad.jsx`.

## Layout y Estructura

### 9. `lang="es"` en `<Html>` del documento base (`pages/_document.jsx`)

**Problema:** El atributo `lang` del elemento `<html>` solo se seteaba vía JavaScript (después del primer render). Los motores de búsqueda y herramientas de accesibilidad necesitan este atributo en el HTML estático.

**Solución:** Se agregó `lang="es"` directamente en `<Html lang="es">`. El script de inicialización ya lo sobreescribe dinámicamente si el usuario elige inglés.

### 10. Viewport y favicon centralizados en `_document.jsx`

**Problema:** El meta `viewport` y el `favicon` se repetían en cada página individualmente, y login/register los tenían omitidos.

**Solución:** Se movieron al `_document.jsx` como defaults globales. Las páginas que necesiten sobrescribirlos pueden seguir haciéndolo localmente con `<Head>` por página.

### 11. Link a `/precios` en el footer global

**Problema:** El footer del `AppShell` no incluía link a la página de precios, obligando a los usuarios a navegar manualmente o encontrarla solo via la landing.

**Solución:** Se agregó el enlace `Precios` en la nav del footer (`components/layout/AppShell.jsx`) entre Blog y Aviso de privacidad.

## Recomendaciones adicionales (no implementadas automáticamente)

Las siguientes mejoras requieren decisión de negocio o trabajo adicional:

1. **Términos de servicio:** Se recomienda crear una página `/terminos` con las condiciones de uso y añadirla al footer y al sitemap. Actualmente solo existe el aviso de privacidad.

2. **CSP (Content Security Policy):** helmet() usa configuración por defecto. Se recomienda configurar una CSP personalizada que restrinja orígenes de scripts/estilos a `'self'` + Google Fonts. Requiere auditar todos los scripts inline.

3. **`sameSite: "strict"` en la cookie de auth:** Actualmente es `"lax"`. Cambiar a `"strict"` elimina el cookie en requests cross-site (incluyendo navegaciones desde links externos), lo que mejora la seguridad pero puede afectar el flujo de redirección post-login desde emails.

4. **Sección de contacto pública:** Existe un endpoint `/api/contact` pero no hay página `/contacto` visible. Se recomienda crear una página de contacto indexable para mejorar la confianza del usuario y el SEO.

5. **Página 404 personalizada:** Verificar que existe `pages/404.jsx` con contenido útil (links a home y proyectos) para mejorar la experiencia cuando los usuarios llegan a URLs inexistentes.

6. **`robots.txt` con URL canónica de Sitemap dinámica:** El sitemap apunta a `https://crawlsite.app/sitemap.xml` hardcodeado. Se recomienda usar `process.env.APP_URL` también en robots.txt (actualmente no es posible en un archivo estático — evaluar generarlo dinámicamente como se hace con sitemap.xml).

7. **Compresión de imágenes OG:** Verificar que `/assets/og-image.png` tenga un peso < 1MB y dimensiones 1200×630px para un preview óptimo en todas las plataformas.
